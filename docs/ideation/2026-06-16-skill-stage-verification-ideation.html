贯穿全篇的核心张力

boss 的设想（4 stage 各做独立 skill + root 编排）几乎就是团队已推翻的 05-21-001 路线——"4 stage 各一 skill + stage runner + 统一 StageSkillResult 协议 + INET smoke" 曾完整实现，2026-06-05 被 06-03-001 整体移除，因为它制造了多层事实源。

所有 survivor 的判别准绳是同一条：收敛到单一 DB 源的「编排元数据 / 验证证据」会复利；离开 DB 源、由 agent 产出再回传的「运行时数据协议」会重蹈覆辙。前者是这次该做的，后者正是上次翻车的。

Codebase Context（grounding）

当前架构：4 stage ["topology","time-sync","flow-template","planning-export"]（scenario-config.ts）。状态机 project-state.ts（locked→current→waiting_confirmation→confirmed）；编排器 agent-adapter.ts runTsnAgent()，stage 流转/确认全在代码骨架。skill 仅 2 个（tsn-topology / tsn-flow-planning），拆分轴是「场景」而非「stage」（主索引 SKILL.md + references/<场景>.md）。MCP 8 个 topology 工具，无任何 INET/语义验证工具（validate 只校验 JSON schema）。time-sync 阶段是 adapter 本地直接写死默认摘要；flow/export 应用层拦截不走 Claude；tsn-inet-export 在 catalog 标 enabled 但磁盘无 SKILL.md（catalog/磁盘 drift）。

既定边界（06-09-002 三层职责）：状态流转 / 阶段顺序 / "必走 MCP 落地" 归 worker 代码骨架（不可交给 skill 文字层——写坏 SKILL.md 只让 agent 空转）；SKILL.md 只承载可调领域指引；落地正确性靠工具白名单（agent 无 Write）+ trusted-signal（只认 MCP 的 summary.mutationId）。06-12-001：按 stage 拆 skill 要新建 3+ SKILL.md，撞上文件粒度播种、Tauri 目录映射、verify-skills 三方对账三条工程链路。

外部生态：INET/OMNeT++（gPTP/TAS/GCL 仿真，但仿真是统计下界、非可调度性证明）· Network Calculus（形式化 WCD 上界证明）· TSNsched（拓扑+流→SMT 合成 GCL→YANG→NETCONF 下发真机）· Siemens PRONETA（verify：compare actual-vs-expected）· CI Quality Gate（verifier 与 generator 解耦防 drift）· LangGraph supervisor（不持业务工具只路由，4-7 specialist 上限）· DO-178C 分级验证（按失效后果分验证证据强度）。

1root 不是 skill，而是状态机的「只读投影」

TOP PICK轴1 编排归属 + 轴3Conf 88%Cx S（决策）/ M（投影实现）
Description
直接回答主题的核心提问"是否需要 root skill 管状态流转"——不需要、也不该是 skill。stage 顺序/状态/可用动作已经是 project-state.ts 的确定性纯函数（createInitial/confirm/requestChanges）。要给 agent 看全局，正确形态是把 WorkflowState 只读投影进 conversationContext（agent-adapter.ts 已有雏形），而非引入一个可写的 root skill。关键区分：编排元数据（编译/注入期消费的静态声明，收敛事实源）vs 运行时数据协议（agent 产出回传，制造事实源）——前者安全，后者正是 05-21-001 翻车点。若团队坚持要"root"，把它严格限制为 LangGraph 式 supervisor：不持任何工具、不写 DB、无持久状态，只读 signal 做路由。
可写 root skill 制造第二事实源（左）vs 只读投影从单一 DB 源派生（右）
Basis
direct: 状态流转已是无副作用纯函数（project-state.ts confirm/requestChanges/recordStageResult）；agent-adapter.ts 已把阶段顺序作为只读上下文注入。reasoned: 再加 root skill 描述同一套顺序 = 两个事实源描述同一状态机，违反 06-09-002「阶段顺序归代码骨架」。external: LangGraph supervisor（不持工具只路由）。
Rationale
这是主题问的那个问题本身。若不把"不该是可写 skill"论证扎实，讨论会默认偏向"加一层"，而那一层正是 06-03-001 刚刻意拆掉的多层结构。
Downsides
"投影"对 agent 是只读，意味着 agent 无法主动改 stage 顺序——若未来真需要 agent 驱动的非线性流转，要另设机制；也需要厘清 boss 说的"root skill"到底指代码层编排器（契合）还是 skill 文字层（冲突）。

2验证 gate 统一为 sidecar /db/<stage>/verify route 族

轴2 + 轴4Conf 85%Cx M
Description
不新建协议层。把每个 stage 的 gate 收敛成 sidecar 的一条 HTTP 路由 /db/<stage>/verify：输入读 DB 当前状态、输出结构化 {ok, errors[], signalId}。新增一个 stage 的 gate = sidecar 加一条路由 + worker allowedTools 加一个工具名，零新抽象。验证结果天然走已有的流式 tool_call 卡片管线（redact + done 对账），用户在聊天里直接看到"time-sync gate：通过 / 3 处冲突"。与被推翻的 StageSkillResult 的本质区别：那个是 agent 产出再回传的"第二事实源"，这个是 sidecar 直接读 DB 算出来的、不离开单一 DB 源。
Basis
direct: topology-tools.ts 的 callSidecarTool 已是 route 化统一形态（8 工具全走 /db/topology/*）；trusted signal = worker 读 value.summary.mutationId。把 verify 做成 route 是同构扩展。external: CI Quality Gate（gate=制品上的幂等读+校验）。
Rationale
"加 time-sync gate 要改几处？"——若答案是"sidecar 一条路由 + worker 一个工具名 + 一行 allowedTools"，复利成立；若要碰协议 type/parse/summarize/apply 四处，就是重蹈 StageSkillResult。这条 route 形状是 4 stage 验证的统一骨架。
Downsides
真正重的 gate（INET 仿真、NC 求解）不是纯 DB 读能算的，需要 sidecar 内接外部进程（见 #5）；route 抽象对"纯结构校验"够，对"跑仿真"是另一类成本。

3trusted-signal 升为状态机硬约束：stage 完成必须带 DB-backed signalId

轴1 + 轴2Conf 84%Cx S-M
Description
当前 confirmCurrentStage 只检查 stage 状态是不是 waiting_confirmation，不检查"这一步到底产出了可信结果没有"。把验证 gate 的阻断语义放进状态机代码骨架：confirm 前先核对该 stage 有没有 trusted-signal（topology 是 mutationId，未来 stage 是各自 gate 的 signalId），没有就拒绝推进，而不是靠 SKILL.md 文字劝 agent。一旦"完成=有 signalId"成为状态机不变量，time-sync 这种现在纯本地写死默认摘要的阶段被迫长出真 gate，验证强度从单个 stage 的特性升级为整个状态机的结构性属性。
Basis
direct: project-state.ts confirm 仅校验 stage.status；topology 路径产 mutationId（agent-adapter.ts topologyMutationId），但 time-sync 路径本地直接 recordStageResult 写死摘要、无任何 DB 背书——两条路径"完成"证据强度不对称。06-09-002「落地靠 trusted-signal」原则正是此意。
Rationale
"time-sync 阶段确认后，DB 里有没有一行能证明它发生过？"——现在没有。把它升为硬约束，约束本身比任何单个 gate 实现都更耐久，且天然抗 SKILL.md 被写坏。
Downsides
没有现成 gate 的 stage（如真机配置）在过渡期会卡住——需要为"外部验证证据到达"设计一个注入 signalId 的人工路径（不是"确认按钮"，而是"证据到位"）。

4验证强度分级（DO-178C 式）挂在「场景轴」，证据跨 stage 复用

轴2 验证 gateConf 80%Cx M
Description
不追求单一 gate，也不为分级引入 root skill。给已有的 ScenarioConfig 加一个 verificationTier 字段：generic-tsn=smoke（结构校验即可）、aerospace-onboard=strict（额外要求 NC 证明 + 真机人工确认）。同一 stage 的 gate 在不同档位跑不同强度。更进一步：低档证据（topology gate 算出的可达性/端口占用矩阵）作为高档验证（flow 调度可行性）的输入被复用，而非每 stage 重算。验证强度随场景走，复用现有场景轴，不新增 stage 轴或 root 编排层。
Basis
external: DO-178C 按失效后果（DAL A-E）分验证证据强度；TSN 流的 criticality（PCP=7 箭载时序 vs PCP=0 尽力而为）就是天然 DAL 等级。direct: ScenarioConfig 已是场景轴单一事实源，加字段零新增文件，避开 06-12-001 拆 skill 撞三链路；dual-plane 场景已对验证有更高隐含要求。
Rationale
箭载场景的确定性时延要求远高于通用场景，一刀切验证强度要么对通用过严、要么对箭载不足。挂在已有场景配置上是改动最小的分级落点；证据复用让验证投资可增量——先给关键 stage 上强档，非关键的保持轻档，不必等 4 stage 全接满仿真才有价值。
Downsides
证据跨 stage 复用要求 gate 输出可被下游消费的结构化中间产物（不只是 pass/fail），增加 gate 返回契约的设计成本；分级策略表本身需要一个权威定义位置（建议挂 ScenarioConfig，不要散落）。

5INET 作 stage 无关的验证后端 + 口径硬隔离

轴4 + 轴2Conf 78%Cx L
Description
建一个 sidecar /verify/inet 通道：把当前 DB 拓扑导成 INET 配置、跑一次轻量仿真、回收 {reachable, gclValid, ...}。通道本身设计成 stage 无关——topology 验结构、time-sync 验同步域可达、flow 验调度可行都复用同一"导出+仿真+回收"管线。但口径必须从数据结构层面严格隔离：把这个 gate 命名为 simulation_loadability_check 而非含糊的 tsn_verify，返回结构强制带 coverage: "loadability_only" 字段——INET smoke 只验 NED/INI 能加载运行，不等于 TSN 可调度性验证（gPTP/TAS/确定性时延都不在内），从源头防止"绿勾=时延有保证"的误导传播到 UI。
Basis
direct: 05-21-001 计划名即 "real-stage-skills-inet-smoke"，e2e/smoke.spec.ts 存在；planning-export 未实现，adapter createSimulationUnsupportedResult 明说没接 runner。external: INET/OMNeT 仿真是统计下界、Network Calculus 才给形式化上界（两者口径差异）；TSNsched 三步 pipeline。
Rationale
最贵的一次性投资（INET 环境、导出器、仿真回收）建好后，每个 stage 的深度 gate 只是换一组断言。而趁 planning-export 还没实现就把命名和返回契约钉死，是最便宜的防误导时机——等接了 INET 再纠名成本高。
Downsides
复杂度最高的一条（需要可运行的 INET/OMNeT 环境 + 跨机 transport，05-21-001 的 SSH/SCP 纪律可复用）；仿真不可复现、是统计下界，"smoke 通过算不算 gate 过"需要配合 #4 的强度分级明确定义。

7time-sync 从「假摘要」变可验证 gate（纯拓扑推导，不需 INET）

Description
time-sync 现在是 4 stage 里"最假"的一环——runTimeSyncStage 直接吐 scenarioConfig.defaults.timeSyncSummary 一句静态文案就等确认，没有任何基于拓扑的验证。最小可验证 gate：用已落库的拓扑算同步域可达性（每个端系统到 GM 候选有路径、同步树无断点）——这是纯拓扑可推导的，不需要先接 INET。让 time-sync 从"假阶段"变成"有真实 pass/fail 的轻 gate"。
Basis
direct: agent-adapter.ts time-sync 产出就是一句静态 timeSyncSummary，无任何拓扑输入；scenario-config.ts 该 summary 是写死的"默认全网已完成时间同步"文案。reasoned: 同步树可达性只依赖已落库的 nodes/links（拓扑阶段已产出），是 4 stage 里除拓扑外唯一无需外部仿真器就能做真实验证的 stage。
Rationale
time-sync 最假，但它的可验证内容恰恰最廉价（纯拓扑推导）。先啃这个，用最低成本证明"每 stage 有可验证 gate"的模式可推广——它是 #2/#3 的最便宜验证落点。
Downsides
可达性只是时间同步的必要非充分条件（真正的 gPTP 收敛/GM 选举/时钟精度要靠 INET 仿真，属 #5 强档）；要避免让这个轻 gate 被误读为"时间同步已验证通过"。
