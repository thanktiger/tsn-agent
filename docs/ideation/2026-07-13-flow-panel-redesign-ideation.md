# 流量规划面板重设计 · ideation

日期：2026-07-13
焦点：把流量规划弹框重设计成「流量列表 / 门控规划 / 软仿模拟 / 硬件部署」平级子 tab（对标时间同步面板），改进多条流在拓扑图上的展示，点某条流出参数详情弹框。视觉由 frontend-design 保证与现有 app 一致。

---

## 先说最重要的一件事：这是「新建」，不是「重排」

截图（含流量列表 F1/F2 行、流详情/编辑弹框、拓扑图红色流路径虚线）是 **docx 规格图/目标设计**，**当前代码里都不存在**。现状 `flow-panel.tsx` 是一张扁平单页，只有三块：

- 门控综合（`plan_tas` → GCL 时序图 `GateTimelineChart` + 明细表 `GclDetailTable`）
- 软仿验证（`verify_tas` → 逐流判决表，这是软仿*输出*不是流清单）
- ST/RC/BE 计数（来自聚合查询 `get_flow_plan`）

流是**经 agent 对话自然语言录入**落库的，面板不逐条回显、没有详情弹框、画布上没有任何流路径高亮（grep `flowPath/highlightFlow/selectedFlow` 在 `src/` 零命中）。

所以本次工作大部分是**从 0 建**：流量列表回显、流详情弹框、拓扑多流可视化都要新写。子 tab 外壳是唯一「重排现有」的部分。

### 两个会卡住的硬约束（需你决策）

1. **完整编辑弹框要扩表。** `flow_streams`（`db.rs:226-245`）没有 MAC / 源目 IP 之外的 MAC 列、VLANid、最早/最晚发送偏移列；抖动是软仿实测输出、不是流入参。截图那份编辑弹框字段**超出当前 DB 模型**，做全量可编辑版要扩 `flow_streams` 表 = 改数据模型，按你的规矩得先确认。
2. **RC 流是双平面双路径。** 802.1CB FRER 的 RC 流复制到 A/B 两个平面（CONCEPTS.md），拓扑可视化不能假设「一条流 = 一条路径」。
3. **flow 域没有硬件部署。** 「硬件部署」当前是时间同步面板独有（`hard-deploy-panel.tsx`）。flow 侧要这个子 tab 也是新增能力（或复用时间同步那套），需明确范围。

---

## 拆成 4 个正交轴

1. **子 tab 信息架构** — 把扁平面板重排成平级子 tab 外壳
2. **流量列表回显** — 逐条流读出来展示（现在拿不到逐行数据）
3. **多流拓扑可视化** — 多条流怎么同时在画布上清楚地画（唯一真正开放的设计题）
4. **流详情弹框** — 点某条流出它的参数

设计一致性（复用时间同步那套子 tab / 命令栏 / 配色）是贯穿 4 轴的约束，不单列。

## 留存者（按建议优先级）

### S1 · 子 tab 外壳：直接复刻时间同步的 `TimesyncSubTabs`（轴 1）
**基础：** `timesync-subtabs.tsx`（42 行）已是成熟范式——`role="tablist"` + `.timesync-subtab` 分段开关，state 提到 App 级经 props 透传、随会话重置。PR #65 还固化了 `computeReveal` 纯函数 + `snapshot.sessionId` 守卫防跨会话残留。
**做法：** 把 `TimesyncSubTabs` 泛化成共享 `PanelSubTabs`（现在有两个消费方了，避免命名漂移），flow 面板配 `["流量列表","门控规划","软仿模拟","硬件部署"]`。子 tab state 提到 App 级、`key={sessionId}` 重挂、复用 `.timesync-commandbar` 布局（分段开关在左、操作按钮在右）。
**为什么值得做：** 这是整个面板的骨架，低风险、有现成范式。是所有其它轴的容器。
**批判后收窄：** 「硬件部署」子 tab 先放一个占位/禁用态（flow 域尚无该能力），不阻塞其余三个子 tab 落地——避免为凑齐 4 个 tab 硬造一个空能力。

### S2 · 流量列表回显 + ST/RC/BE 区分（轴 2）
**基础：** 后端现在只有聚合 `get_flow_plan`，前端拿不到逐条流。`flow_streams` 有全部列，`flow_route.rs` 能算路由但没接前端。
**做法：** 新增一个 Rust 查询命令（类似 `get_flow_plan` 但返回逐行 rows：流 id、talker→…→listener 路径、名称、class、pcp、period_us、frame_bytes、max_latency_us）。前端列表按 class 三态视觉区分（ST=确定性、RC=双平面冗余、BE=尽力而为，PCP 固定 7/6/0），每行一个「详情」按钮。
**为什么值得做：** 没有列表回显就没有详情入口、也没有画布高亮的选中锚点。是 S3/S4 的前置。
**批判后收窄：** 节点路径要 `flow_route` 出「流→途经链路序列」并映射到 edge——这是 S2 和 S3 共用的后端能力，一次做好。

### S3 · 多流拓扑可视化：选中高亮起步，渐进到并行车道（轴 3，皇冠）
这是唯一真正开放的设计题。外部调研收敛出清晰答案，且 Cisco/Kiali/Grafana 都在多路径这块「摆烂」（各自只画单路径或聚合成一条边）——做出来就领先。

**分阶段（对齐你「改动最小」偏好）：**
- **第 1 步（最小可用）：** 点流量列表某行 → 画布只高亮那一条流的路径（RC 则高亮 A/B 两条）。就是截图那根红虚线，但由「选中」驱动。复用 `CHART_COLORS` 上色，扩 `tsn-floating-edge` 的 className。改动最小、直接满足截图行为。
- **第 2 步：** 画布旁加流图例（复用图表库通用交互：hover 某流 → 其余淡出 opacity 0.2 + 该流加粗；点击持久隔离）。
- **第 3 步（可选）：** 多流同屏并行车道——共享链路上把各流按序号横向偏移成平行色带（ggraph `geom_edge_parallel` 的 `offset=(i-(n-1)/2)*spacing`）。`tsn-floating-edge` 已有平行边分槽逻辑可扩。2–10 条流可读。

**明确否决：**
- 边捆绑（FDEB/Edge-Path Bundling）——调研明说只有 >10–15 条流才需要，我们 2–10 条属过度工程。
- 聚合边 + 百分比（Kiali/Grafana 模型）——那是「关心流量*大小*」的场景；我们是「保留每条流*身份*」的 PCB 式需求，聚合会抹掉流身份。

### S4 · 流详情弹框：只读起步，可编辑推后（轴 4）
**做法：** 点「详情」→ 弹框展示该流现有参数（class、pcp、周期、帧长、帧数、talker/listener 路径、最大延迟、redundant）。复用 app 现有弹框范式。
**批判后收窄：**
- **只读版**用 `flow_streams` 现有列，零 schema 改动，能立刻上。
- **完整可编辑版**（截图那份 MAC/VLAN/发送偏移）需扩表 + 「编辑→重新规划」链路，且引入**第二条写路径**（流现在只由 agent 录入）——碰 app 的 agent-native 对等原则（UI 能改的 agent 也要能改）。**推后，flag 给 boss 决策**：要不要扩表、UI 编辑是否要镜像成 agent 工具。

## 被否决 / 降级的想法（附理由）

- **为「硬件部署」子 tab 现造一个 flow 侧硬部署能力** → 降级为占位。范围过大、且未确认是否复用时间同步那套；不该为凑 4 个 tab 造空能力。
- **一次性把截图整套（含可编辑弹框）实现** → 拆分。撞 DB 模型缺字段 + agent-native 第二写路径两个决策点，硬做会先改数据模型再返工。
- **默认多流全同屏并行车道** → 降为第 3 步可选。第 1 步「选中高亮」已满足截图行为且改动最小；并行车道等确认要「所有流同屏对比」再上。
- **自研边捆绑算法** → 否决（见 S3）。

## 设计一致性（frontend-design 接手点）

选定方向后交给 frontend-design，产出子 tab 外壳 + 流量列表 + 流图例 + 详情弹框的视觉，复用现成件：`TimesyncSubTabs` 分段开关、`.timesync-commandbar` 布局、`CHART_COLORS`（8 色色盲友好）、`PanelCta` 居中 CTA、`GateTimelineChart` 零依赖 SVG 手法、`key={sessionId}` 重挂 + 会话守卫。

**WebKit 坑（PR #65/#97 踩过）：** 子 tab 容器用 flex-wrap 不用 `grid+overflow`（会塌行）；图标-only 的 `aria-label` span 要加 `role="img"` 否则 CI a11y 卡。
