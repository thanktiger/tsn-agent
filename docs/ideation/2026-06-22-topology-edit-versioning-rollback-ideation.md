# Ideation：拓扑编辑的版本 / 回退能力

起因：现在 agent 改拓扑（initialize / apply_operations / 挪位置）**没有版本概念**。用户说「回退上一步 / 上几步」时，大模型做完意图判断后，是**按它的理解去「尝试恢复」**——不确定、可能越改越偏。boss 抛了两个方向：DB 记录版本化、DB 文件用 git 管。

**需求证据先打个问号**：现在「用户会频繁要求回退多步」还是一句假设，没有真机会话日志、投诉或事故支撑。而且有个零成本基线没被对比过——用户本就在对话里，可以直接说「把你刚加的那条链路删掉」，让 agent 再走一次 apply_operations。如果真实场景大多是**单步**回退，对话重述可能已经覆盖大半价值；持久化版本库恰恰是为最缺证据的「多步 / time-travel」建的。进 ce-brainstorm 前，建议先从 dev/真机日志捞一小撮样本，确认回退请求的频率与跨步数，再决定值不值得建持久化层。

## 现状 grounding（实地核实，非臆测）

- `topology_mutation_buffer.rs`：内存 ring buffer（1024 上限、**进程重启清零**），每条只存 `{session_id, domain, mutation_id}`，**不存拓扑状态**。它是「变更通知 / 补增量」机制，不是版本库。
- DB（sqlx/SQLite）的 `topology_nodes` / `topology_links` **只有当前态**，没有任何 history / snapshot / version 表。`tx.rollback()` 全是 SQL 事务回滚（错误处理），跟用户级「回退」无关。
- `apply_operations` 收的是**结构化原子 ops**（`link_delete` / `node_add` / `link_add` / `node_update` / `node_delete`），且每次提交挂一个**单调 mutationId**。
- **但「挪位置」不走这条路**：`update_node_position` 是裸 `UPDATE topology_nodes SET x,y`，不经 apply_operations，却照样 push 一个 mutationId。所以 mutation 流里有「有 id、无可重放结构化 op」的空洞——这点对 S2/S4 是硬约束（见下）。
- `next_id` 是内存里的全局计数器，**进程重启从 0 重新计数**，不是 per-session、也不持久化。

**一句话**：结构化 ops + 单调 id 是个不错的底子，但**有两个缺口**——挪位置不在 ops 流里、mutationId 重启就清零，**而且完全没有持久化的状态历史**。所以「事件溯源已就位 80%」是高估了；真要做，持久化层得自带一套独立的、跨重启单调的版本序，不能直接复用 mutationId。

## 核心洞察（贯穿所有想法）

1. **「回退」该是确定性操作，不是大模型恢复**——和确认按钮 / request_stage_change 同一套哲学。大模型只判断**意图**（回退到哪一步 / 上 N 步），回退动作本身确定性执行。但要注意：「确定性」目前只对 DB 这一层成立，运行时还有 mutation_buffer、画布拖拽位置、会话指针、**以及对话中 agent 自己的认知**没被覆盖（见 S1）。
2. **历史该存在 DB 里，不是 git 管二进制 / 导出文件**——boss 的「git 式历史」直觉对，但落点错（见否决 R1/R2）。
3. **mutationId 不能直接当持久化版本键**——它是内存全局计数器、重启归零。持久化日志/快照表必须用自己的、跨重启单调的版本序（如 DB 生成的 per-session `version_seq`）；启动时若想复用 mutationId，得先从 DB 里该 session 的 `max(version)` 恢复 next_id，否则重启后新 id 与旧版本行碰撞（同 session 两个「版本 5」指向不同状态）。

## 存活的想法（排序）

### S1（头牌）确定性回退工具 + UI 按钮，把「回退」从大模型猜测变确定操作
大模型只判意图「回到版本 N / 上一步」，真正回退由确定性的 `topology.rollback(toVersion)` 执行（MCP 工具 + 画布上一个「撤销」按钮，agent-native 对等）。S2/S3 是它的底层存储。

**这里最难的不是 DB 还原，是回退后的状态再同步**，必须在 brainstorm 钉死，否则「确定性」只是个口号：
- **mutation_buffer**：UI 靠 `get_topology_mutations_since` 拉增量。DB 退回版本 N 后 buffer.next_id 仍停在 now，增量对账会错乱——回退后要么重置 buffer，要么让 UI 强制全量 refetch + 重置 last_seen。
- **画布位置**：节点 x/y 是 drag 持久化的独立状态。回退按快照还原会覆盖用户回退后手动挪的位置，按 ops 重放又会漂移——得明确 x/y 在回退里的去向（随快照走 / 保留当前 / 提示用户）。
- **对话中的 agent 认知**：这是最隐蔽的坑。用户说「回退两步」→ DB 确定性退回，但 agent 上下文还记着它刚做的那两步、「相信」那些节点链路还在。下一轮「在刚加的交换机上再连一条线」就会引用已被回退掉的 sync_name → `references unknown source node`。这恰好是开篇控诉的「越改越偏」，只是换了触发路径。**回退完成必须给 agent 注入一条权威状态通告，或强制 next-turn re-inspect**，否则确定性回退只是把脑补从「恢复动作」挪到了「回退后的下一次编辑」。

**依据**：direct——现状回退靠 LLM「按理解恢复」；而 confirm 按钮 / request_stage_change 证明本仓偏好确定性操作。
**为什么重要**：把不可靠的「脑补恢复」换成可信的一键回退，是这条线的全部价值所在——但价值能否兑现，取决于上面三处运行时同步是否一起做掉。

### S2 事件溯源：把结构化 ops 持久化成 append-only 日志
新建 per-session 的 mutation 日志表，按独立 `version_seq` 把每批 ops 原样落库（append-only，永不改写）。回退 = 清空当前态后从 0 重放到 N，或对 N+1..now 应用逆 op。

**重放正确性不是免费的**，S2 必须先满足这几个不变量（这正是 S3 用简单性换掉的成本）：
- 重放**必须先清空当前态**再从 0 放，否则 `node_add` 撞 PK（`topology_nodes` PK=`(session_id, sync_name)`，apply 路径是裸 INSERT 无 ON CONFLICT）。
- `initialize` 是整表 DELETE+重建，它在日志里是个**重放断点**——0..N 里只要夹了一个 initialize，前序日志失效，重放起点要重置到该 initialize。
- `node_update` 必须**同时存旧值**，否则逆 op 无从生成。
- **挪位置不在 ops 流里**（见 grounding），要么把它也建模成 `node_update` op 落日志，要么承认 S2 无法无损覆盖位置（只有 S3 快照能）。

**依据**：reasoned——apply_operations 已是结构化原子 ops，结构贴现状；但「已挂单调 id 可直接复用」是错觉（id 重启清零）。
**为什么重要**：append-only ops 日志能精确重放/逆向，满足回退目标。（audit、「改了什么」、time-travel 调试**不在本期目标内**——别拿它们当 S2 优于 S3 的选型理由，那是范围扩张。）

### S3 快照式：每次 mutation 存一份拓扑全量（推荐作 MVP 底层存储）
每次 apply/initialize 后，把当前 nodes/links 复制进一张版本表，打上独立 `version_seq`。回退 = 直接还原快照 N。
**依据**：reasoned——比事件溯源简单（不担心重放正确性、不踩 PK/initialize/旧值那一串坑）；拓扑就几十节点，全量快照存储成本微不足道。
**为什么重要**：最稳、最好实现的 MVP；用确定性还原换一点存储，几乎零风险。文档自己已给它「最稳」定性——所以推荐方向就**直接选 S3**，S2 留到「ops 日志确有新需求」时再单独 ideation。

### S4 先给「看见改了什么」：确定性 diff
即便先不做完整回退，也可以暴露两个版本间的确定性 diff（加了/删了哪些节点链路），让 agent 和用户看清改动。
**依据**：direct——大模型「按理解恢复」的根因之一是看不到精确变更；让 agent 先看见 diff，可能就把恢复从「脑补」变「有据」。
**前置依赖要说清**：diff 需要能取到**两个时点**的拓扑状态，当前 DB 只有当前态，diff 无对象可比。所以 S4 **不是可以先于回退能力独立先发的廉价品**——它依赖 S3（快照表）或 S2（可重放）先落地，落地后才「几乎免费」。它的价值是 compounding 衍生增量，排期上跟在持久化层之后，别当 MVP 第一步。

## 否决的想法（及原因）

- **R1 git 管 DB 二进制文件**（boss 直觉之一）：SQLite 是二进制——git 存它无法 diff/merge、whole-file churn、锁冲突。否。boss 的「git 式历史」直觉对，但要在 **DB 内**做（S2/S3），不是 git 管二进制。
- **R2 git 管「DB 导出文件」**（boss 另一直觉）：导出是**派生物不是真相源**；要维护「改→导出→commit→回退→re-import」整条链 + git 进程开销 + 双源一致性风险。比 in-DB 版本重且绕。否——就「回退」这件事而言成立。**但**它指向的「可 git 协作的拓扑源文件」是**另一个产品方向**（团队协作 / PR 评审拓扑），那条线是否更高杠杆，取决于目标用户是否在团队里协作设计拓扑——建议跟单机回退一起纳入需求信号收集，别默认当噪音丢掉。
- **R3 直接拿 mutation_buffer 当版本源**：它是内存 ring、**进程重启清零、只存 id 不存状态**——不能作版本存储，必须新建持久化层（且持久化层用独立版本序，不复用其 next_id，见核心洞察 3）。
- **R4 引外部版本控制系统 / 新依赖**：过度工程，DB 内就能解决。
- **R5 逐键 / 逐字撤销粒度**：编辑粒度是「一次 agent op 批次（一个版本）」，更细无意义。
- **R6 撤销/重做逆 op 栈（命令模式）做成独立方案**：单独看很诱人（细粒度 undo/redo），但**不独立成立**——`initialize`=整表重建的逆推不出、跨重启持久化是难点、挪位置不在 ops 流里也无逆 op。它最多是 S2/S3 之上的一个交互层，不是与 S1/S3 并列的候选。若 S3 落地后真要 redo，再单独 ideation。

## 推荐方向

**S1（确定性回退）做骨架，S3（快照表）做底层存储**，**不碰 git 管二进制/导出**。S2 留作「ops 日志确有新需求」时的后续 ideation，S4（diff）是持久化层落地后的衍生增量、不抢第一步。

但在投实现之前，先把两件事各自收口：

1. **需求证据**：从真机/dev 日志确认回退请求的频率与跨步数，并与「让用户再指示 agent」这个零成本基线对比。若证据只支持单步回退，范围可进一步收到「单步撤销 + diff」，连持久化版本库都未必要建。
2. **定位自觉**：版本存储 + 画布「撤销」按钮，是把产品从「对话式生成」往「带历史的可控编辑器」推一步。这是个有意识的复杂度下注（多一张表、多一个 MCP 工具、UI 与 agent 工具双面维护），不是顺手做的，得想清谁受益、对新用户心智模型的影响。

需求证据站得住、定位下注认了，再对 **S1 + S3** 走 ce-brainstorm，钉死：
- **版本键**：用独立 `version_seq`，重启时从 DB 恢复 next_id（核心洞察 3）。
- **回退的运行时同步范围**：DB + mutation_buffer + 画布 x/y + **回退后强制 agent re-inspect/注入状态通告**（S1 三处坑），不止 DB。
- **回退是新 mutation 还是真删历史**：倾向**追加新版本**（保 append-only/单调/可再回退「回退」本身），「真删 N+1..now」会破坏 append-only、让 redo 失效、版本号出空洞——别当未定细节。
- **回退入口**：chat 意图（大模型判版本号）、画布按钮、还是两者都要，MVP 先定一个。
- **UI 与 agent 工具如何对等**。
