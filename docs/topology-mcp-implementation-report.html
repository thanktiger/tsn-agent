MCP Usage Guide

TSN Topology MCP 使用指南

这份文档面向外部用户、MCP 客户端和其他大模型，说明如何独立启动并调用 tsn_topology。该服务是面向 TSN 拓扑阶段的确定性 stdio MCP server，只接收结构化参数，返回 IntermediateTopology 摘要或受控 full payload。

服务名 tsn_topology
版本 0.1.0
传输 MCP stdio
工具 8 个工具
快速开始 接入方式 工具参考 数据边界 调用示例 错误与限制 验证方式 模型规则

一、快速开始

1

安装依赖

在仓库根目录执行 npm install，确保 Node 运行时和 MCP SDK 依赖可用。

2

构建服务

执行 npm run build:worker，生成 src-node/dist/tsn-topology-server.mjs。

3

配置 MCP 客户端

让客户端通过 node 启动打包后的 stdio server，而不是手工向终端输入 JSON。

4

先查询模板

模型或客户端必须先调用 topology.describe_templates，再按返回的当前 catalog 生成 topology.initialize 参数。

本机 stdio MCP 配置示例

{
  "mcpServers": {
    "tsn_topology": {
      "command": "node",
      "args": [
        "/ABS/PATH/tsn-agent/src-node/dist/tsn-topology-server.mjs"
      ]
    }
  }
}

直接运行检查

cd /ABS/PATH/tsn-agent
npm run build:worker
node src-node/dist/tsn-topology-server.mjs

看到 tsn_topology MCP server listening on stdio 表示服务已启动。该进程等待 MCP 客户端通过 stdio 通信，普通终端下不会进入交互式命令行。

二、接入方式

Human / Model

用户或大模型

负责理解自然语言、选择模板、生成结构化参数和决定是否需要用户澄清。

MCP Client

客户端 / Bridge

支持 MCP 的客户端可直接 spawn stdio server；普通 LLM API 需要 bridge 把 function call 转成 MCP tool call。

Transport

MCP stdio

当前实现使用 StdioServerTransport，适合本机子进程集成和确定性集成测试。

MCP Host

tsn-topology-server.ts

注册 topology.* 工具，统一返回 JSON 文本形式的 tool result。

领域核心

src/topology/*

确定性拓扑模板、校验、inspect、artifact 和 operations 逻辑。

调用链路：用户/模型 -> MCP client 或 bridge -> tsn_topology stdio server -> topology domain -> JSON tool result

支持 MCP 的客户端

配置 mcpServers.tsn_topology 后，客户端会自动启动进程、发现工具并把工具结果交还给模型。

普通大模型 API

模型本身不会直接连 MCP。需要业务 runtime 暴露 function schema，再由 runtime 使用 MCP client 调用 tsn_topology。

远程调用

当前没有 HTTP transport。跨机器共享时应新增 Streamable HTTP host，并补鉴权、请求限制和日志脱敏。

三、工具参考

模板

topology.describe_templates

返回当前可用模板目录、参数约束、默认值和示例。模型必须以实时返回为准，不得凭文档示例猜模板。

初始化

topology.initialize

用结构化 templateId 和 params 生成 IntermediateTopology。

查询

topology.inspect

按 node/link selector 查询节点、链路、邻接和端口占用摘要。

摘要

topology.describe_artifacts

在调用方已经持有 artifact 对象时，汇总数量、大小、节点链路和 MAC 条目数量。

校验

topology.validate_intermediate

校验 IntermediateTopology schema、节点、端口、链路和规模限制。

构建

topology.build_artifacts

从中间拓扑构建四份 JSON artifact，默认只返回摘要。

校验

topology.validate_artifacts

在调用方已经持有 artifact 对象时，校验 artifact schema 和跨文件引用关系。

编辑

topology.apply_operations

P0 支持 link.delete、node.add、link.add 原子操作。

接口 关键入参 默认返回 full 返回限制
topology.initialize templateId、params、responseMode 模板 ID、节点数、链路数、交换机数、端系统数 仅当 responseMode: "full" 且显式设置 topologyFullAllowed: true 时返回 full topology；这不是鉴权
topology.inspect topology、selectors、includeAdjacency 选中节点、选中链路、邻接摘要 Agent-facing MCP 禁止 full portUsage 外发
topology.build_artifacts topology artifact 计数、名称、总字节数、MAC 条目数 Agent-facing MCP 禁止 full artifact 外发
topology.apply_operations topology、operations、dryRun 节点链路数量和 changeSet 计数摘要 只允许返回 updated topology；完整 changeSet 会被剥离

工具命名

MCP 原始工具名是 topology.*。部分 Agent SDK 会把它映射成 fully-qualified 名称：

mcp__tsn_topology__topology_describe_templates
mcp__tsn_topology__topology_initialize
mcp__tsn_topology__topology_inspect
mcp__tsn_topology__topology_describe_artifacts
mcp__tsn_topology__topology_validate_intermediate
mcp__tsn_topology__topology_build_artifacts
mcp__tsn_topology__topology_validate_artifacts
mcp__tsn_topology__topology_apply_operations

四、数据模型与边界

中间拓扑模型

  • schemaVersion 固定为 tsn-agent.topology.intermediate.v0。
  • 节点类型支持 switch、endSystem、server；项目层 bridge 当前主要消费 switch 和 endSystem。
  • 链路介质 P0 固定为 ethernet，链路速率支持 10、100、1000、10000 Mbps。

本地保留数据

  • 完整 artifact、端口占用表、MAC 转发表、full changeSet 只进入本地 domain、app-runtime、project/session storage 和导出层。
  • 诊断日志只保存 run id、session 状态、chunk 统计、artifact 路径和错误摘要。
  • 对话和执行步骤不保存凭证、完整敏感上下文或大段原始工具输出。

明确不做

  • 不做自然语言意图解析，NL 到结构化参数由 Agent / project 层完成。
  • 不生成 mac-forwarding-table.html，没有 topology.render_mac_table_html。
  • 不生成完整 project，不推进四阶段 workflow，不直接写仓库文件，不保存长期 topology handle。
数据 请求侧是否应传入 MCP 响应侧是否返回给模型 说明
IntermediateTopology 可以。inspect、validate_intermediate、apply_operations 需要它。 仅 initialize / apply_operations 在 full opt-in 时返回。 这是 topology MCP 的主数据模型。
artifact 对象 外部模型侧不建议传入；只有本地 runtime 已持有 artifact 时才调用 describe_artifacts / validate_artifacts。 不返回 full artifact；build_artifacts 只返回 summary。 纯 MCP 主链路从 topology 出发时，使用 build_artifacts 的摘要即可。
端口占用表 / full changeSet 不作为外部模型输入。 不返回；full changeSet 会被剥离。 只在本地 domain 或 runtime 内部使用。
模板 ID 状态 用途 给模型的规则
generic-line 当前可用 通用线型 smoke、基础拓扑生成。 可按当前 catalog 使用；会暴露简化参数 endSystemsPerSwitch。
generic-ring 当前可用 通用环形 smoke、基础冗余拓扑生成。 可按当前 catalog 使用；会暴露简化参数 endSystemsPerSwitch。
dual-plane-redundant 当前可用 通用双平面/双归属/双冗余拓扑。 必须传入显式 planes、switches、switchGroups、endSystems 和接入关系；不要传 endSystemsPerSwitch 等数量 shortcut。

示例 A：从 0 初始化拓扑

1 用户输入

自然语言需求

例如“生成 4 个交换机、每台接 2 个端系统的双平面冗余拓扑”。

2 Agent 层

查询模板

先调用 topology.describe_templates，读取可用模板和参数约束。

3 MCP

初始化

调用 topology.initialize，传入当前 catalog 支持的结构化参数。

4 MCP

内部校验

topology.initialize 内部校验 schema、节点、端口、链路和规模限制。

5 Client

保存 full

需要继续编辑时，请求 full topology 并保存在客户端侧。

6 App

合成项目

TSN Agent 项目层可把 topology 合成项目草稿；MCP 不做 project 生成。

当前实现的预期摘要

上述初始化应得到 switchCount = 4、endSystemCount = 8、nodeCount = 12、linkCount = 18。这里没有 endSystemsPerSwitch：模型或项目层需要先把自然语言中的数量描述展开成 8 个明确端系统及其主备接入关系。

参数展开规则

Agent / Project 层负责

  • 理解“4 个交换机、每台接 2 个端系统”等自然语言。
  • 选择 dual-plane-redundant 模板。
  • 生成明确的交换机、交换机组、端系统 ID 和 A/B 主备接入关系。
  • 必要时向用户澄清端系统分配、平面互联和跨平面链路策略。

MCP 层负责

  • 校验必填字段、引用完整性、A/B 平面一致性和规模限制。
  • 确定性分配端口、坐标和链路 ID。
  • 相同输入始终返回相同拓扑或相同错误。
  • 拒绝 switchCount、endSystemsPerSwitch、endSystemCount 这类双平面 shortcut。

模板边界

generic-line / generic-ring 仍暴露 endSystemsPerSwitch，这是基础 smoke 模板的简化参数。dual-plane-redundant 不接收该字段；MCP 只执行结构化参数校验和确定性生成，不从自然语言或数量 shortcut 推断端系统分配。

示例 C：校验当前拓扑

{
  "tool": "topology.validate_intermediate",
  "arguments": {
    "topology": { "...": "IntermediateTopology" },
    "responseMode": "summary"
  }
}

成功时返回 ok: true 和节点、链路、错误数量摘要；失败时返回结构化错误数组。模型应使用 path 定位要修正的字段，不要让 MCP 猜测缺失含义。

确认后重放

dryRun 默认只需要 summary，用于展示影响摘要。用户确认后，客户端用同一份 topology snapshot 和同一批 operations 再调用一次；如果后续工具需要 updated topology，再在确认后的重放请求中使用 responseMode: "full" 和 topologyFullAllowed: true。

六、错误模型与限制

类别 当前限制
节点数量 maxNodes = 256
链路数量 maxLinks = 1024
单节点端口数 maxPortsPerNode = 64
单次操作数 maxOperations = 32
artifact 总字节数 maxArtifactBytes = 1,000,000
JSON 深度 maxJsonDepth = 32
入站 payload maxIngressPayloadBytes = 1,000,000

统一错误 envelope

{
  "ok": false,
  "errors": [
    {
      "code": "LIMIT_EXCEEDED",
      "message": "...",
      "path": "$.operations",
      "severity": "error",
      "details": {},
      "retryable": false,
      "requiresUserClarification": false
    }
  ],
  "warnings": [],
  "metadata": {
    "responseMode": "summary",
    "summaryOnly": true
  }
}

禁止未 opt-in 或敏感 full payload

FORBIDDEN_RESPONSE_MODE 用于拦截未显式 opt-in 的 full topology 请求，以及 full artifact、端口表或完整 changeSet 等敏感或大体积 payload。设置 topologyFullAllowed: true 不是鉴权，只是调用方显式声明需要 full topology；当前信任边界是能否启动并访问该本机 MCP 进程。

P0 不支持完整 CRUD

node.delete、node.update、link.update 当前返回 UNSUPPORTED_OPERATION。

可要求用户澄清

AMBIGUOUS_SELECTOR、UNKNOWN_TEMPLATE_ID 等错误会通过 requiresUserClarification 指示交互层。

七、验证方式

Build

构建服务

执行 npm run build:worker，确认生成 stdio server bundle。

Unit

跑 MCP 测试

执行 npm test -- src-node/mcp/topology-tools.test.ts，检查 registry、schema 和 tool result 边界。

Boundary

验证响应边界

确认默认 summary、显式 opt-in full topology、拒绝 full artifact、剥离 full changeSet 的行为。

Client

接入 MCP 客户端

用客户端配置启动 node src-node/dist/tsn-topology-server.mjs，调用 topology.describe_templates 做冒烟测试。

当前打包状态

当前实现支持 Node stdio dev/test host。Tauri 生产 resources 暂未携带 topology MCP host；如果要随桌面应用打包，需要单独设计固定路径、签名/hash 校验、private IPC 或 localhost-only、会话 capability token 和 fail-closed 策略。

八、给模型的调用规则

先发现，再调用

不要凭记忆或本文目标示例猜模板字段。每次新建拓扑前先调用 topology.describe_templates，只按实时返回的 template catalog 生成参数。

不要传自然语言

自然语言只在 Agent/project 层解析。MCP 只接收结构化 JSON 参数，例如 templateId、params、topology、selectors、operations、dryRun、responseMode 和 topologyFullAllowed。

full payload 要显式 opt-in

只有确实需要继续传给后续工具时，才请求 responseMode: "full" 和 topologyFullAllowed: true。这不是安全鉴权；真正的 capability token 属于未来生产 sidecar 设计。

编辑前先消歧

如果用户说“交换机 1”，先用当前项目上下文或 topology.inspect 解析成稳定 ID。命中不唯一时先问用户，不要提交歧义 selector。

dryRun 后再应用

已有拓扑编辑应先 dryRun: true 获取影响摘要；用户确认后，用同一份 topology snapshot 和同一批 operations 重放。

错误按 path 修复

收到错误时读取 code、path、requiresUserClarification。缺字段、端口冲突、节点不存在都应修正参数后重试。
