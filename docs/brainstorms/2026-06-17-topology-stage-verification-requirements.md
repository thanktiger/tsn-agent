# 需求：拓扑阶段验证（本地结构校验 + 接 INET 真验）

> 一句话：给拓扑阶段加「验证」——用户点「确认并继续」时先自动查拓扑对不对，有问题就拦住、用一句人话说哪里不对、修好才放行；所有结论都标明「验到哪一级」，杜绝"绿勾=时延有保证"的误读。分两批交付：第一批本地、不依赖 INET、能独立验收；第二批接远端 INET 真跑。

---

## 问题陈述

拓扑画完后，软件几乎不知道它"对不对"——现有唯一检查只看了"节点数 &gt; 0"和"有没有连到空气的线"（`validate` 工具）。用户（boss，TSN 网络规划者）无法在进入下一阶段前知道拓扑是否结构完整、是否能被 INET 真跑、转发是否成立。本需求为拓扑阶段补上一道"过关闸"，并为后续每个阶段的验证铺一条可复用的通道。

源自点子梳理（origin_ideation），boss 已确认其中 7 个方向全做、分两批。

---

## 角色

- **A1 规划者（boss）**：用 TSN Agent 配置 TSN 网络的人，看不懂工程术语；要的是"我这拓扑能不能用"的一句话答复 + 哪里不对。
- **A2 助手（agent）**：在对话里驱动各阶段；本需求中验证由代码骨架触发，agent 不自行编造验证结论。
- **A3 sidecar（应用后端）**：读数据库算验证结论的唯一权威；验证闸沿用既有信任模型（一条路由 + 一个工具 + mutationId 同款的可信信号）。

---

## 需求

### 通用（两批共用）

- **R1**：验证由用户点「确认并继续」触发，作为拓扑阶段进入下一阶段的**过关闸**——不通过则拦住、不推进。
- **R2**：每个验证结论都带**口径标签**，表明验到哪一级：`仅结构级` / `仅能加载运行` / `已验可调度性`（后者本期不做、但标签体系要占好位）。"绿"只代表所标那一级，不等于时延/可调度性已保证。
- **R3**：失败时给**一句中文结论 + 具体问题清单**（指明哪个节点 / 哪条链路出错），让 boss 能直接动手修；原始技术细节收进可展开处、不强塞给用户。
- **R4**：验证结论经 sidecar 读库计算后、走既有可信信号通道回到应用层；不重建"让大模型回传结构化结果"的协议（那是 05-21-001 被删的原因）。

### 第一批：本地结构校验（不依赖 INET、断网可用）

- **R5 结构检查**：sidecar 直接读库（`topology_nodes` / `topology_links`，节点以 sync_name 标识）做图论校验，规则集起步包含：连通性、端口两两配对、链路对称、无孤立节点、交换机/端系统角色合法、编号（sync_name/linkSeq）无重复。规则以"可扩展清单"形式组织（后续加规则=加一条，不改引擎）。
- **R6 MAC 转发表现算现验**：复用 `topology_compute.rs` 已有 BFS（`build_legacy_mac_forwarding_table`）现算转发表，校验"每个目的恰好一个出端口、无转发环、所有端系统全可达"。算完即弃、**不落库**。
- **R7 MAC 纳入过关闸**：拓扑若算不出合理转发表（有目的不可达 / 成环），判结构不过关、拦住推进。MAC 表本身现在**不展示、不外发**（规划阶段开启后再发给规划器）。
- **R8 第一批口径 = `仅结构级`**：第一批的"绿"标 `仅结构级`，明确未跑任何仿真。

### 第二批：接远端 INET 真验

- **R9 序列化成 INET 包**：把库内拓扑导成一个 inet-bundle —— `tsnagent/generated/network.ned`（`package tsnagent.generated`，TsnSwitch/TsnDevice/EthernetLink/`ethg++` 连接）+ `omnetpp.ini`（最小可加载：`sim-time-limit`、`cmdenv-interactive=false`、`cmdenv-express-mode=true`）+ `manifest.json`（schema `tsn-agent.export-manifest.v0`）。此包验证与未来导出共用同一布局。
- **R10 节点类型→INET 模块映射**：交换机→`TsnSwitch`、端系统→`TsnDevice`；模块命名 `sw{N}` / `es{sw}_{n}`（从库内 sync_name + node_type 推导）。任何类型映射不出来视为验证错误、在发 INET 前就报。
- **R11 远端加载 smoke**：把包发到远端 INET 主机、跑 `inet -u Cmdenv -f omnetpp.ini -n .`，以退出码判定"能否加载运行"；结论口径标 `仅能加载运行`，明确不代表时延/调度已验。
- **R12 转发表对账**：让 INET 自己从拓扑生成转发表，与我方 BFS 算的逐条对比（terraform-plan 式 diff），不一致处精确定位到"交换机×目的"的出端口。
- **R13 阶段无关通道**：远端验证通道做成不绑定拓扑阶段，后续 time-sync / flow 阶段可复用同一"导出+发送+跑+回收"管线。

## 关键流程

- **F1 第一批·确认即验（过关闸）**：A1 在拓扑阶段点「确认并继续」→ sidecar 读库跑结构检查(R5)+MAC现算现验(R6/R7) → 全过：标 `仅结构级`、一句"结构没问题"、放行进下一阶段；有错：拦住、列出问题(R3)、`仅结构级` 下不放行，A1 修完再点确认。
- **F2 第二批·接 INET 真验**：通过第一批后（或单独触发）→ sidecar 把拓扑序列化成 inet-bundle(R9/R10) → 经远端通道(R13)发到 INET 主机跑加载 smoke(R11) → 回收退出码 + INET 自生成转发表，对账(R12) → 一句结论 + `仅能加载运行` 标签 + 差异清单。

## 验收示例

- **AE1**（结构错被拦）：拓扑里 SW3 有一条链路只连了一端 → 点「确认并继续」→ 被拦，提示"交换机 SW3 有一条线没接对端，拓扑不完整"，不进下一阶段。Covers R1/R3/R5。
- **AE2**（孤立端系统被拦）：某端系统没连任何交换机 → 被拦，提示该端系统未接入。Covers R5。
- **AE3**（转发不可达被拦）：拓扑分裂成两个互不连通的子网 → MAC 现算发现有目的不可达 → 判结构不过关、拦住。Covers R6/R7。
- **AE4**（通过）：合法的星型/线型拓扑 → 点确认 → "结构没问题（仅结构级，未验时延）" → 进入时间同步阶段。Covers R1/R2/R8。
- **AE5**（口径不被误读）：任何通过的结论都带 `仅结构级`/`仅能加载运行` 标签，界面不出现脱离标签的裸"绿勾"。Covers R2。
- **AE6**（INET 能加载）：第二批，合法拓扑导成包发到远端 INET → `inet … omnetpp.ini` 退出码 0 → "能在 INET 加载运行（仅能加载运行级）"。Covers R9/R10/R11。
- **AE7**（转发对账出差异）：我方 BFS 与 INET 自生成转发表某条出端口不一致 → 结论列出该"交换机×目的"差异，不是只给红叉。Covers R12。
- **AE8**（类型映射缺失先报）：出现无法映射到 INET 模块的节点类型 → 发 INET 前就报 `unmappable_node_type`，不产坏 NED。Covers R10。

## 范围边界

### 本期做
- 第一批：R5/R6/R7/R8 + 通用 R1-R4，卡在「确认并继续」上。
- 第二批：R9-R13，接 boss 的远端 INET 主机。

### Deferred for later（以后做）
- **MAC 表发给规划器**：现算的转发表当前不外发；规划/导出阶段开启后再接。
- **更高级别验证**：gPTP 收敛 / TAS·GCL 调度 / 可调度性（NC 上界 / SMT）——`已验可调度性` 标签先占位，本期不实现。
- **远端主机可配置化**：本期对接 boss 那台固定主机；多主机/凭据管理后续。

### Outside this product's identity（不属于本产品）
- 不在桌面应用内打包 OMNeT++/INET（重型原生依赖，始终跑在外部主机）。
- 不替代形式化验证工具（NC/SMT）；INET 是仿真、是统计下界，本产品只如实标注口径、不冒充可调度性证明。

## 依赖与假设

- **远端 INET 已就绪（已真机验证 2026-06-17）**：Ubuntu 24.04 / INET 4.6.0 / OMNeT++ 6.4.0，经 Tailscale `100.104.38.106`（用户 zhang）；启动器 `~/.local/bin/inet`（opp_env+nix）；配方 `cd <bundle> && inet -u Cmdenv -f omnetpp.ini -n .`（现场重跑 EXIT=0）。成功样例 bundle 在远端 `~/tsn-agent-inet-verify/`。
- **NED 契约**：`docs/ned-contract.md` 描述的 NED/INI 契约仍适用、被真机证实。
- **库已"去 Qunee 化"**：节点以 sync_name 标识、无 imac/sync_type 杂质（PR #35），结构校验与 INET 命名推导从更干净的库出发。
- **现算转发表能力已存在**：`topology_compute.rs::build_legacy_mac_forwarding_table`（BFS），读内存拓扑、不读库。
- 跨机传输（SSH/SCP）与远端进程管理沿用 05-21-001 留下的纪律：结构化 argv、超时、进程组杀、密钥脱敏。
