你关心的结论

拓扑 MCP 是确定性服务

只接收版本化 IntermediateTopology、初始化参数和结构化 operations，运行固定 initializer、validator、inspect 和 artifact 逻辑。

P0 先做可测闭环

P0 覆盖模板发现、初始化、校验、artifact 和插入交换机 tracer subset；完整 CRUD 和行业模板进 P1。

Project 不放进 topology MCP

generate_project、flow、时间同步、NED/INI 和仿真导出留在 project/simulation 层。

不导出 MAC 表 HTML

topology.render_mac_table_html 不进入服务，也不再生成 HTML artifact。

review 后补强的边界

host 与 domain 分离

P0 先定义 topology domain 和 MCP adapter/dev host；tsn-mcp-host 是生产集成阶段候选形态。

权威 schema

IntermediateTopology 必须版本化，表达节点、链路、端口和模板元数据，并覆盖现有样例 fixture。

生产打包是决策门

stdio、Tauri-managed host、共享 package + adapter 需要先比较，再进入生产 sidecar 和 Node-free 验收。

安全默认收紧

生产方案必须明确进程所有权、固定随包路径、完整性校验、资源上限、summary/full 边界和 full 数据本地清理。

端到端链路

1

用户输入

用户用自然语言描述新拓扑或修改当前拓扑。

2

Agent 解析

Agent/project 判断是无拓扑初始化，还是已有拓扑上的增量 operations。

3

初始化或消歧

无拓扑时先发现模板并形成初始化参数；已有拓扑时解析稳定节点/链路 ID。

4

Topology MCP

执行 describe_templates、initialize、inspect、apply_operations、validate。

5

Project composition

通过 bridge 把 updated topology 和 changeSet 合成 canonical project，并判断 flow 是否需要重新规划。

6

客户端展示

展示拓扑变化、工具状态、确认提示和后续阶段入口。

Topology MCP 工具面

工具 用途
topology.describe_templates 返回模板 ID、参数约束、默认值和适用场景标签，供 Agent/Project 层选择模板。
topology.initialize 从 templateId 和结构化参数生成 initial intermediate topology。
topology.inspect 只读查询节点、链路、端口占用、邻接关系和引用摘要。
topology.apply_operations P0 对当前 intermediate topology 原子应用插入交换机所需 operations，P1 扩展完整 CRUD。
topology.validate_intermediate 验证节点、链路、端口和引用完整性。
topology.build_artifacts 生成兼容 JSON artifact，不直接写用户目录。
topology.validate_artifacts 校验 legacy artifact 之间的一致性。
topology.describe_artifacts 返回 artifact 计数和诊断摘要；不返回完整 artifact 原文。

两条拓扑路径

No Topology

从 0 初始化

project/Agent/skill 层先调用 topology.describe_templates，再提取或确认模板 ID、交换机数量、终端数量、链路速率和冗余策略，调用 topology.initialize。

Existing Topology

增量修改

project/Agent 层先把当前 canonical/legacy 拓扑转成 IntermediateTopology，用 topology.inspect 完成 ID 消歧，再把已解析 operations 交给 topology.apply_operations。

为什么需要 IntermediateTopology

不是语义中间态

它不是让大模型理解用户意图的草稿，而是 topology domain 的版本化事实模型。

不承载项目身份

topologyId 和 revision 属于 project/client 层；MCP P0 不维护服务端拓扑状态，调用方每次传入完整 topology。

支撑可测试工具链

validate、apply_operations、build_artifacts 和 fixture 都以它为输入输出，避免模型上下文或 canonical project 成为第二事实来源。

确定性验证规则

先验证，再应用

整批 operations 作为原子事务处理；任一操作失败，都不返回半应用后的拓扑。

字段必须齐全

新增节点/链路必须包含 ID、类型、端口/布局策略、链路两端、介质和速率等关键项。

不做猜测

不得靠显示名猜节点；非稳定 selector 或命中不唯一时返回歧义错误。

引用必须完整

链路两端节点必须存在；更新和删除必须指向稳定 ID。

端口必须无冲突

同一节点端口不能被多条链路占用；删除链路会释放端口。

输出必须稳定

不生成随机 ID、随机坐标或时间戳；相同输入得到相同 topology 和 changeSet。

预览必须可确认

dryRun 返回 changeSet；确认时客户端重放同一份 topology snapshot 和 operations，若当前拓扑已变化则重新预览。

资源必须有上限

节点、链路、端口、operation、JSON 深度、artifact 大小和执行时间超限时返回结构化错误。

响应与数据边界

Agent-facing 默认 summary

只返回摘要、计数、warning/error summary 和必要字段；歧义错误只给候选数量和可继续消歧的摘要。

full 只留在本地

完整 artifact、端口占用、MAC 表和 full changeSet 只进入 project/client 层；full topology 仅在授权后作为结构化工具链输入输出。

MCP 不保存长期句柄

服务端不维护长期 artifact handle；本地缓存由 project/client 层生成、清理，不能进入 Agent 上下文或诊断日志。

实际案例：从 0 初始化一个环形拓扑

关键点

无当前拓扑时不会调用 topology.apply_operations。Agent/Project 层先发现模板并形成结构化初始化参数，MCP 用 topology.initialize 生成 initial IntermediateTopology。

用户

自然语言

“新建一个 4 个交换机、每个交换机 2 个终端的环形 TSN 网络。”

发现模板

describe_templates

Agent 查询可用模板，确认 generic-ring 支持交换机数量、终端挂载和链路速率参数。

初始化

initialize

调用方提供模板 ID 和数量参数，MCP 返回 initial topology。

合成项目

Project bridge

project 层把 intermediate topology 合成 canonical project，后续时间同步和 flow 规划继续留在项目层。

实际案例：在交换机 1 和 2 之间添加交换机 3

关键点

这句话不会直接进入 MCP。Agent/project 层先把“交换机 1/2/3”解析成稳定 ID 和结构化 operations，MCP 基于传入的 current topology 验证和执行。

用户

自然语言

上游

消歧

交换机 1 -> sw1，交换机 2 -> sw2；当前拓扑由 project 层转换为 IntermediateTopology。

MCP

预览

用 dryRun 验证 link.delete + node.add + 两个 link.add，返回 changeSet。

结果

确认应用

用户确认后重放同一份 topology snapshot 和 operations，并返回 updated topology 与 changeSet。

Tauri 打包形态

  • P0 先提供开发/测试可运行的 topology domain 和 MCP adapter。
  • P0 只要求 domain fixture 与 project bridge 最小 round-trip；完整 staged workflow 集成进入 P1/integration。
  • 生产前比较 stdio packaged server、Tauri-managed host、共享 package + adapter 三条路径。
  • 如果选择 stdio，通常由 Agent SDK/MCP client spawn server；不要假设同一个 stdio 进程同时被客户端和 Agent 共享。
  • time_sync.*、flow.*、simulation.* 属于后续独立需求。
  • 只有出现独立生命周期、不同运行时或隔离需求时，才拆成多个 MCP server。
  • 生产包若承诺不要求用户自行安装 Node，必须同时覆盖现有 Agent worker 链路。

常用拓扑模板

  • P0：generic-line、generic-ring，以及可由现有规则 fixture 化的 aerospace-redundant 场景模板。
  • P1 候选：industrial-line、industrial-ring、automotive-zonal、aerospace-dual-plane 等行业模板。
  • 模板通过 topology.initialize 输出 initial intermediate topology，不输出完整 project，也不输出一串 add operations。
  • Agent/LLM 通过 topology.describe_templates 查询可用模板，再由 Project/Agent 层选择或向用户确认。

内容校验

校验项 结果 说明
是否区分无拓扑初始化和已有拓扑修改 Pass 无拓扑走 topology.initialize，已有拓扑才走 topology.apply_operations。
是否覆盖节点/链路增删改查模型 Pass 已明确“查”走 topology.inspect，P0 “增删改”覆盖插入交换机 subset，P1 再扩完整 CRUD。
是否保留自然语言到落地架构链路 Pass 保留“从 0 初始化环形拓扑”和“交换机 3 插入交换机 1/2 之间”两个摘要案例。
是否避免过细 JSON / 错误码矩阵 Pass 已移除大段 schema、返回样例和长错误码矩阵。
是否明确 MCP 不调用大模型 Pass 职责边界和确定性规则中均已体现。
是否解耦 generate_project、NED/INI、HTML 导出 Pass 明确列为不属于 topology MCP。
是否覆盖 Tauri 打包和多 server 策略 Pass 已区分 P0 dev/test adapter 和生产架构决策门，并标记 Node-free 全链路风险。
是否覆盖 review 后的高风险点 Pass 补充权威 schema、dryRun 预览、summary/full、本地缓存清理、资源上限和敏感 artifact 边界。
