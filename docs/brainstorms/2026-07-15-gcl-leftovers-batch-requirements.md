# GCL 遗留四项收口

## 摘要

GCL 功能线合并后的四项遗留一批收口：软仿路径钉死的可行性 spike、流量面板规划流程重做（求解器选择 + 两步重新规划 + stale 禁验证）、flow_verify_command 拆分还债、castup 对外格式文档交付。四项相互独立、各自交付。

## 背景

门控明细表/详情弹窗/路径指定全线已 MERGED（PR #116–#119）。遗留四项 boss 已逐项拍板：spike 做实验验证、stale 禁用按钮并重做规划入口交互、拆分走 ce 流程、格式文档输出给 boss。

## 需求

### R1 KTD13 spike：软仿逐跳转发钉死（实验，非产品功能）

- 目标：验证「用户指定的绕路路径能否在 INET 软仿中逐跳钉死」——当前指定绕路的流规划按绕路算门控，但软仿验证流量仍走 INET 自算最短路，验证结果不代表真实部署。
- 场景：同平面三交换机绕路拓扑，宿主机（bfe_agent@100.125.25.12）真机验证。
- 已知边界（FRER spike 证据）：INET 路由 xml 配置只覆盖 talker 的 L3 出口，管不到中间交换机的 L2 转发——spike 要回答的正是中间跳用什么机制钉住。
- 产出：可行性结论（可行/不可行 + 机制 + 证据）沉淀 `docs/solutions/`；**产品化另起周期**，本批不改产品代码。
- 判据：指定绕路的流在软仿中实际按绕路转发（观测手段由 plan 定，如 .sca/.vec 或转发表 dump）。

### R2 求解器选择 UI

- 流量面板规划初始态在「规划门控表」按钮上方增加求解器选择区，设计语言对齐时间同步软仿的覆盖参数区。
- 选项两个：`inet-z3`（默认选中、可用）；`open-planner`（**置灰不可选，标「预留」**，为后续调度规划 API 对接留位）。
- 数据落位：复用 `flow_gcl_plan` 既有 provider 维度（当前常量 `inet-z3`），表结构零改动；open-planner 接入时只补执行逻辑。

### R3 重新规划改两步流程

- 点「重新规划」不再立即重跑：回到初始态（求解器选择 + 「规划门控表」按钮），再点按钮才执行规划。
- 交互镜像时间同步「重新仿真」（回初始态、参数可重选）。

### R4 stale 时禁用验证按钮

- `stale=1` 时「软仿验证」按钮禁用；既有琥珀提示条（"流量参数已变更，建议重新规划门控表"）保留。
- 重新规划成功（stale 复位）后按钮恢复可用。

### R5 flow_verify_command 拆分（纯重构）

- 走 ce-plan 出拆分方案后执行；行为零变化，以既有测试全绿为准；独立 PR。

### R6 castup 对外交付文档

- 基于 `docs/solutions/flow-planning/gcl-windows-format-v1.md`（单表 + 裸数组路径 + raw 文件终态）整理对外交付版，MD + HTML 双格式，产出路径交给 boss。
- 是否投 castup.cn 由 boss 决定，默认不投。

## 验收

- AE1：spike 在真机跑通绕路场景并产出结论文档（可行/不可行 + 证据）。
- AE2：无规划结果的工程进流量面板：可见求解器选择（inet-z3 选中、open-planner 置灰标预留）→ 点「规划门控表」→ 正常出结果。
- AE3：有结果态点「重新规划」→ 回到选择态且不立即重跑；再点「规划门控表」才执行。
- AE4：改流参数触发 stale → 「软仿验证」按钮禁用 + 琥珀条在；重新规划成功后按钮恢复。
- AE5：拆分后 cargo + vitest 全绿，验证行为与拆分前一致。
- AE6：交付文档 MD + HTML 生成，boss 拿到本地路径。

## 不做

- open-planner 的实际 API 调用、baseUrl 配置项（纯占位预留）。
- 多 provider 结果并存/对比展示（当前仅 inet-z3 产生数据）。
- 钉死机制产品化（待 R1 spike 结论，另起周期）。

## 交付切分

四项独立：R1 spike 无产品代码（实验 + solutions 文档）；R2–R4 同属流量面板一个 PR；R5 独立重构 PR；R6 文档交付无 PR。
