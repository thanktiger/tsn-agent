# 三类流 ST/RC/BE 混跑（Qbv + CB）需求

## Summary

流量链路从「仅 ST」扩展成 ST/RC/BE 三类混跑（pcp 固定 7/6/0）：ST 走 Z3 门控不变；RC 是 802.1CB FRER 冗余流——录入时按双平面推导 A/B 两条不相交路径，软仿里源端帧复制、目的端帧消除，覆盖「帧消除去重」与「单路故障」两个验证态（老需求 R17 留后续的 FRER 逻辑本期落地）；BE 尽力而为。pin 仿真时从 ST 门表推导 BE/RC 门的互补关窗，防止低优先级流量占端口压垮 ST。验证判据按类分级。

---

## Problem Frame

单 ST 流全链路已通（录流→Z3 真算 GCL→软仿实测三项全绿），但真实组网是三类流混跑，这条链路有三块欠账。其一，混流没有保护：真机实证 BE 走恒开门会持续占端口，ST 窗开时端口正在发 BE 帧（非抢占），ST 反复错窗——时延 688ms、丢一半。其二，`docs/prototypes/TSN典型组网测试方案_20260527.docx` 的用例 3/6（802.1CB 冗余传输）一直没覆盖：schema 与双平面路由在前作已备好（`redundant`/`paths` 槽、plane 路由），FRER 逻辑本身留的后续周期就是现在。其三，验证判据是全部流一把尺（收=实发、抖动<1us、时延≤窗口），BE 在灌流争用下必丢包、RC 的通过标准是去重与容错——一把尺对它们是误判。

---

## Key Decisions

**RC = 802.1CB FRER 冗余流，不是限速流。** 对话中曾走向 TTE 的 rate-constrained 语义（CBS 信用整形），对照 docx 用例 3/6 后修正：RC 指帧复制与消除——源端复制到双平面 A/B 两路、目的端去重，通过标准是「单路故障零丢包」与「重复帧正确消除」。CBS/ATS 等整形类因此无需求支撑，不做。

**混流保护 = pin 阶段互补关门后处理，Z3 只喂 ST。** pin 仿真时从 ST 的 GCL 按端口推导 BE/RC 门的补集窗：ST 开窗期间关、其余时间开；BE/RC 门保持 INET 隐式保护带默认开（帧发不完不许进窗，天然防跨入 ST 窗），ST 门维持已修的关隐式保护带（Z3 零余量窗需要）。不把 BE/RC 塞进 Z3——BE 无周期语义建模失真，且约束面膨胀。这是行为变化：现在全部流都进 Z3 configuration，改后只 ST 进。

**验证判据分级。** ST 三项不动；RC 判去重后收=实发、断链仍零丢包（docx 通过标准原文），时延/抖动报告不判；BE 只判连通（有收包），其余指标报告不判。「BE 不干扰 ST」不设独立判据，由 ST 自身达标体现。

**单路故障验证靠软仿断链，健康/故障两态都做。** docx 的帧复制验证靠分别断开两个发送口；软仿对应为运行中断掉一条平面的链路。有 RC 流时一次验证触发覆盖三轮：健康态一轮 + A/B 各断一轮；只验去重不验容错等于 FRER 只做一半，不取。

**class↔pcp 固定映射。** ST=7、RC=6、BE=0，录入校验强制，不允许自定义 pcp。三类天然分门（gate_index=pcp 已就绪），Z3 排 gate7、gate6/gate0 走互补关窗。

## Key Flows

F1. **录入 RC 流。** **Trigger:** 用户在对话里描述 RC 流。→ 校验闸在既有各项之上增加：class↔pcp 映射匹配、按双平面推导 A/B 两条不相交路径（节点/链路不相交断言）。推导不出双路径（单路径/线性拓扑）→ 响亮拒绝并说明需双平面；通过 → 落库（`redundant`、双路径填入预留槽）。**Covers R1, R2, R3.**

F2. **规划。** **Trigger:** 用户触发"规划"。→ 只把 ST 流喂给 Z3 综合 GCL；RC/BE 不参与排窗。无 ST 流时产出「无需门控」的合法空结果。落库形态与 pin 语义沿用。**Covers R4, R5, R6.**

F3. **软仿验证（健康态）。** **Trigger:** 用户触发"验证"。→ pin ST GCL + 按端口推导 BE/RC 门互补关窗 + RC 流装配帧复制/消除 → 跑软仿 → 分级判据逐流判决：ST 三项、RC 去重后收=实发且无重复、BE 有收包；报告附 gPTP 收敛诊断行。**Covers R7, R8, R10, R11, R12, R13, R15.**

F4. **软仿验证（故障态）。** 有 RC 流时随同一次触发追加两轮：分别断 A、B 平面各一条在途链路（运行中断开），断链时刻在流量前半程（约仿真时长 30%–50% 处，保证断后仍有可观帧量须经单平面送达，容错验证不空转）；故障轮只判 RC 判据，ST/BE 判据仅在健康轮生效；无 RC 流不跑故障轮。**Covers R9, R12.**

## Requirements

**录入与校验**

R1. 录入支持三类 `class`，pcp 固定映射 ST=7/RC=6/BE=0，校验闸拒绝不匹配的组合。
R2. RC 流录入时按双平面推导 A/B 两条不相交路径（节点/链路不相交，断言失败即拒），并写入 `redundant`/`paths` 预留槽；推导不出双路径的拓扑上 RC 录入被拒并说明原因。
R3. ST/BE 的录入与校验行为不变（回归保障）。

**规划（门控综合）**

R4. Z3 configuration 只含 ST 流；RC/BE 不排窗。
R5. 无 ST 流的流集，规划产出「无需门控」的合法结果；验证不再以空门控表硬拦。
R6. 规划结果落库与 pin 消费形态不变。

**软仿装配**

R7. RC 流装配 802.1CB 帧复制/消除：源端复制注入 A/B 两路、目的端消除去重（INET FRER 机制，模块组合规划期定；宿主机 `framereplication` 与 `combiningfeatures/frerandtas` showcase 为模板）。
R8. pin 时从 ST GCL 按端口推导 BE/RC 门互补关窗：ST 开窗期间关、其余开；BE/RC 门保持隐式保护带默认开，ST 门维持关隐式保护带。无 ST 窗的端口各门恒开。
R9. 软仿支持故障注入：运行中断开指定链路，且该能力在现有 bundle/HTTP 服务形态内自包含（不加新服务接口）；断链选择须避开 ST 流路由与 gPTP 同步树所经的链路，无法避开时响亮报告而非静默跑出误判。
R10. gPTP 与 BE 共 pcp0 门，须在软仿中验证时钟收敛不因两条路径劣化：互补关窗的扣帧、BE 灌流下的 pcp0 队列争用/溢出（混合场景与纯 BE 场景为必测）；判据沿时间同步阶段 offset 阈值；若劣化，给 gPTP 独立通道作为备选方案。

**验证判据与场景**

R11. ST 判据不变：收=实发±在途容差、稳态抖动<1us（跳首样本）、时延≤max_latency（缺省 period）。
R12. RC 判据分两态：健康态判去重后收=实发±在途容差（容差按 A/B 较长一路计，「无重复帧」保持严格）；故障态（A/B 各断一轮）判仍零丢包（同容差口径）。时延/抖动报告不判。
R13. BE 判据：有收包即过；时延/抖动/丢包报告不判。
R14. 新增 ST+RC+BE 同拓扑争用的混合验证场景：BE 灌满剩余带宽（沿前作 AE5 语义，涓流不算数），三类各按各的判据，ST 不因混流劣化。
R15. 每次流量验证报告附 gPTP 收敛诊断行（复用 R10 的 offset 阈值检查，只报告、不参与流量判决），使关窗或队列争用导致的时钟劣化直接可见，而非以 ST 抖动 FAIL 的形式间接暴露。

## Acceptance Examples

AE1. **Covers R7, R12.** RC 双路健康：发 10000 → 目的端去重后收=实发±在途容差、无重复帧 → PASS；出现重复帧或缺口超容差 → FAIL。
AE2. **Covers R9, R12.** RC 断 A 平面（流量前半程断链）：去重后收=实发±在途容差 → PASS；缺口超容差（真丢包）→ FAIL。断 B 平面同理。
AE3. **Covers R2.** 5 跳线性拓扑录 RC 流 → 拒绝，提示需双平面拓扑（docx 线性场景亦无 CB 用例）。
AE4. **Covers R8, R14.** 混合场景 BE 灌满剩余带宽：ST 三项仍绿（互补关窗生效）、RC 两态过、BE 有收包 → PASS；ST 劣化 → ST 判 FAIL（暴露保护缺陷，不得靠放宽判据染绿）。
AE5. **Covers R5, R13.** 纯 BE 流集（无 ST）：规划出「无需门控」、验证正常跑、BE 按连通判。

## Scope Boundaries

- CBS/ATS/帧抢占等整形类不做——RC 语义修正后无需求支撑。
- 录入面板 UI 不做，沿前作决策走对话录入；面板挪后续 PR。
- 802.1CB 之外的过滤类特性（802.1Qci per-stream filtering）不做。
- 硬件真机 CB 部署不做——软仿为界，T10 硬件判决在产品之外。
- RC/BE 的时延、抖动只报告不判，不为其发明经验阈值。

---

## Dependencies / Assumptions

- 宿主机 INET 4.6 具备 FRER 能力面：`showcases/tsn/framereplication` 与 `combiningfeatures/frerandtas`（FRER+TAS 组合）均在（2026-07-03 核实），INET 侧语法有官方模板可抄。
- 断链注入假设 INET 现成机制（如 ScenarioManager）可在 ini/ned 文件内自包含表达——规划期 spike 验证，是 R9 的前提。
- 双平面拓扑与 plane 路由已有（dual-plane 生成器、`derive_route` 带 plane 参数）；`topology_streams` 的 `redundant`/`paths` 槽已在库（前作 R4）。
- gPTP 被互补关窗短暂扣帧（每门控周期微秒级）预期不影响收敛；BE 灌流下的 pcp0 队列争用是更强的潜在干扰——两者由 R10 软仿验证 + R15 逐次诊断兜底。
