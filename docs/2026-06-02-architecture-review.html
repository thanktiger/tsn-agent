核心数据流

一句自然语言 → App.submitIntent → agent-adapter.runTsnAgent → (web 走 fake-agent / 桌面走 Tauri invoke → Rust → Node worker → Claude SDK + 拓扑 MCP) → TsnAgentResult 三态 union → project-state 归一化 → artifact-bundle 汇总 → 用户确认 → project-exporter / project_writer 落盘。

4 个稳定阶段 ID（topology / time-sync / flow-template / planning-export）+ record → waiting_confirmation → confirm 的状态机是整个项目最硬的约束 — AGENTS.md 明确禁止改成场景专属 ID。

2优点 — 值得保留

① 分层清晰、无循环依赖

app → agent → domain 单向流，export / topology / planner 各成纯业务域。

② 类型契约硬边界

4 个 stage ID、TsnAgentResult 三态、ArtifactBundle、ExportManifest 都是稳定契约，AGENTS.md 兜底。

③ 核心代码大量纯函数

domain/、topology/(12)、export/(7 exporter) 均 input→output，335+ vitest 通过。

④ Fail-closed 设计

web 直接 CTA 下载桌面版；桌面端 sanitizer + 词表 redactor + 16KB/200/30/90s 上限严格。

⑤ 安全网关到位

project_writer.rs 拒绝 root/home/symlink、atomic write；display-redaction 剔除 Claude 字样。

⑥ Release 工具链自动化

prepare-release.mjs 同步 5 个文件版本；verify-skills.mjs 是 build:worker 的守门人。

⑦ 文档与计划同步推进

brainstorms / plans(18) / adr 三层递进；契约文档与代码同步。

⑧ 测试可重放

agent-result-fixtures.ts + agent-stage-dispatcher.ts 替代 fake runtime，刚清理 868 行测试黑盒。

3缺点 — 按严重度排序

问题 位置 影响
1 App.tsx 上帝文件 src/app/App.tsx · 2188 行 · 40+ state · 14 直接 import 多 agent 并行改主 UI 必冲突；planner 轮询/导出/diagnostics/阶段确认全混。
2 commands.rs 上帝文件 src-tauri/src/commands.rs · 749 行 IPC + spawn + stdout 解析 + 路径查找 + 校验全混。
3 worker 单文件 55KB src-node/claude-agent-worker.mjs SDK 初始化 / 拦截 / 提取 / 审计 / prompt 拼装一锅烩。
4 IPC 协议无版本号 三端 JSON line Worker event 是字符串枚举，stage_results: Vec<Value> untyped；schema 漂移无早期检测。
5 类型契约无单一来源 前端 TS / Rust serde / Node JSON 三套手对齐 改一处忘改另一处，靠 review 兜底。
6 agent/ 命名重叠 agent-adapter · fake-agent · workflow-stage-result · topology-workflow-stage-result · stage-skill-contract 5 个相似名字，新协作者迷路。
7 export 分层混乱 artifact-bundle(汇总+容器双职) / project-exporter(协调) / project_writer(写盘) 三层重叠，加新 exporter 要改 artifact-bundle。
8 tsn-topology/ 游离 独立 .git、未提交、不在 root tracking 被 .claude/skills + src-node/mcp 引用，三处可能漂移。
9 PR 无 CI .github/workflows/ 仅 production-build.yml 在 release/** 触发 main 与 PR 推进无自动验证，多 agent 协同时合并冲突隐匿。
10 Planner 轮询内嵌 App App.tsx 826–896 行 · 8 个 helper · 手管 timeout ref 应抽 hook / service。
11 diagnostics 侵入业务 App.tsx 调 logDiagnostic 10+ 次 业务和监控耦合。
12 集成测试薄 1 个 Playwright smoke；4 个 #[ignore] 真 SDK e2e MVP 够用，规模化不够；缺 Rust↔worker IPC e2e、MCP stdio e2e。
13 黑盒兼容代码 sessions/session-topology-repair.ts 违反 boss "拒绝冗余兼容代码"规则。
14 缺 CLAUDE.md 单一入口 当前靠 AGENTS.md + README + memory 拼图 新 session 上手成本偏高。

4演进建议 — 按目标排序

P0 立即做 · 拆解多 agent 协同瓶颈

  1. 拆 App.tsx 为 hooks（最高优先）： useTsnSession() · useStageWorkflow() · usePlannerPoller() · useArtifactExport() · useDiagnostics()。 App.tsx 只剩 ~250 行 layout，每个 hook 独立测、被独立 agent 拥有。
  2. 加 PR CI workflow：on: pull_request 跑 npm test + cargo:test + tsc --noEmit + verify:skills + e2e:ui-smoke。
  3. 决定 tsn-topology/ 角色：3 选 1，不要继续"游离"。 (a) 已迁完直接删；(b) 转 git submodule 锁版本；(c) 转 npm workspace 真复用。

P1 一个 sprint 内 · 结构清理

  1. 拆 commands.rs：worker_bridge.rs 管 spawn + stdout 行解析；commands.rs 只剩 tauri handler + 校验。
  2. 拆 claude-agent-worker.mjs： sdk-adapter.mjs / event-bridge.mjs / result-extractor.mjs / prompt-builder.mjs 四份。
  3. IPC 协议加版本号：所有 worker event envelope 加 protocol_version: "1.0"，三端共享 shared/ipc-schema.ts + zod。
  4. agent/ 契约收敛：workflow-stage-result + topology-workflow-stage-result + stage-skill-contract 合到 src/agent/contracts/；adapter / sanitizer / dispatcher 与 contracts 分开。
  5. export 三层归一：exporters/*.ts 单格式纯函数；bundle.ts 只组装数据；project_writer 只做 bundle → 盘；删 project-exporter 中间层。

P2 持续 · 可测试 + 协同

  1. 共享 schema 单一来源：zod 定义 agent event / stage result / artifact / session payload；Rust 端用 ts-rs 反向同步或 JSON Schema 派生。
  2. 集成测试补强： (a) Rust spawn worker JSON 往返（不依赖 Anthropic）； (b) MCP stdio e2e； (c) 多阶段确认 / 快速路径 / 旧 session 迁移 三个 Playwright 用例。
  3. diagnostics 反转依赖：业务发事件，独立 logger 订阅，不在业务代码里散调。
  4. 删 session-topology-repair（或写明 sunset 日期，按 boss 规则不留双轨）。
  5. 加 CLAUDE.md：常用命令 + 不可逾越约束 + 当前迭代 PR + 必读契约文档清单。

P3 远期 · 规模化

  1. 轻量状态库（Zustand）替 App 40 state。
  2. exporter 插件机制，迎接其余 7 个应用场景。
  3. tauri-driver / Playwright desktop 真实 e2e，把 4 个 #[ignore] 测试拉进 nightly。

5一句话总结

当前架构骨架对了（分层清晰、契约稳定、纯函数为主）， 问题集中在三个上帝文件（App.tsx / commands.rs / claude-agent-worker.mjs） + 三套手对齐的类型契约。 先按 P0 三件事下手，多 agent 协同立刻解锁； P1 把上帝文件拆完，从 MVP 到真实项目的演进路径就清晰了。

建议起点

从 "拆 App.tsx 成 hooks" 开始 — 收益最大、风险最低：纯重构、测试已覆盖主流程、改动单点不影响契约。 完成后 App.tsx 从 2188 行降到 ~250 行，5 个 hooks 各自独立可被不同 agent 拥有。
