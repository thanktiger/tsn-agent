# Schema 草案 — Plan v3 U1 入口 deliverable

## ⚠️ 重大发现：4 件套与 plan v3 / brainstorm v3 假设不一致

参考 CDT 工程 `docs/analysis/json-generation-analysis.html` 的实际 JSON 生成链路，确认 TSN planner 输入实际是 **4 件套**：

| Plan v3 / brainstorm v3 R1 假设 | CDT 实际 | P0 内 |
|---|---|---|
| `topology.json` | `topology.json` | ✓ |
| `topo_feature.json` | `topo_feature.json` | ✓ |
| `data-server.json` | **❌ 上游源**（Qunee 编辑器存档，不是 planner 输入） | — |
| `mac-forwarding-table.json` | **❌ CDT 不生成此文件** | — |
| (未列) | **✓ `node.json`**（节点详细配置 9 类业务 + base_info） | ✓ P0 |
| `flow_plan_1.json`（推 P1） | `flow_plan_<id>.json` | ✗ P1 |

**正确 4 件套 = topology.json + topo_feature.json + node.json + flow_plan_<id>.json**

**P0 范围（沿用 brainstorm v3 "P0 = topology only + 停 flow-dependent 导出"决策）** = 3 件套：
- `topology.json` 对应表 (topology_nodes / topology_links / topology_refs)
- `topo_feature.json` 对应表 (topo_feature_links)
- `node.json` 对应表 (nodes + 9 类业务配置子表 / JSON 列)

**P1 范围**：
- `flow_plan_<id>.json` 对应表 (flow_plans / flow_plan_streams / flow_plan_paths / flow_plan_node_paths)

**Plan v3 R1 / brainstorm v3 R1 必须修正**（在本 schema 草案 confirmed 后）：删除 `data-server.json` / `mac-forwarding-table.json`、加入 `node.json`。

```

  ├── convertDataToTopoFeature() → topo_feature.json
  └── refreshNodeMap() → node.json (骨架 base_info_cfg)

用户在"流规划"页填表
  ↓ flowPlanSave (flowPlan.ts:70)
  └── flow_plan/flow_plan_<id>.json

下游派生：
  node.json + topo_feature.json
    → convertNodeJSONToNewNodeJSON
    → new-node.json (推北向接口/烧录硬件)
```

## 表结构草案

通用约定：
- 所有表 `session_id TEXT NOT NULL` + `REFERENCES sessions(id) ON DELETE CASCADE`
- 时间戳列用 INTEGER (Unix ms)
- 复杂嵌套（数组/对象）走 JSON TEXT 列
- 主键内嵌 `session_id` 防跨 session 引用
- 所有索引以 `(session_id, ...)` 开头

### 1. `topology.json` → 3 张表

#### `topology_nodes`
```sql
CREATE TABLE topology_nodes (
  session_id        TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  imac              INTEGER NOT NULL,             -- Qunee 内部唯一 ID
  sync_name         TEXT NOT NULL,                -- node display name (数字字符串)
  x                 REAL NOT NULL,                -- 画布 X
  y                 REAL NOT NULL,                -- 画布 Y
  sync_type         TEXT NOT NULL,                -- icon resource path (Q.Graphs.exchanger)
  node_type         TEXT NOT NULL CHECK (node_type IN ('switch', 'networkcard', 'T10')),
  PRIMARY KEY (session_id, imac)
);
CREATE INDEX idx_topology_nodes_session ON topology_nodes(session_id, sync_name);
```

#### `topology_links`
```sql
CREATE TABLE topology_links (
  session_id        TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  link_seq          INTEGER NOT NULL,             -- 同 session 内自增 (排序用)
  name              TEXT NOT NULL,                -- "0:0-1:0" 格式
  src_imac          INTEGER NOT NULL,             -- 源节点 imac (from._ref)
  dst_imac          INTEGER NOT NULL,             -- 目的节点 imac (to._ref)
  left_label        TEXT,                         -- styles.leftLabel
  right_label       TEXT,                         -- styles.rightLabel
  speed             INTEGER,                      -- styles.speed (Mbps)
  macrotick         INTEGER,                      -- styles.macrotick (写时往往空)
  styles_extra      TEXT,                         -- 其余 styles 字段 JSON
  PRIMARY KEY (session_id, link_seq),
  FOREIGN KEY (session_id, src_imac) REFERENCES topology_nodes(session_id, imac),
  FOREIGN KEY (session_id, dst_imac) REFERENCES topology_nodes(session_id, imac)
);
CREATE INDEX idx_topology_links_session ON topology_links(session_id, src_imac, dst_imac);
```

#### `topology_refs`
```sql
CREATE TABLE topology_refs (
  session_id        TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  ref_key           TEXT NOT NULL,
  ref_value         TEXT NOT NULL,                -- 原样保留 Qunee 反向引用
  PRIMARY KEY (session_id, ref_key)
);
```

### 2. `topo_feature.json` → 1 张表

#### `topo_feature_links`
```sql
CREATE TABLE topo_feature_links (
  session_id        TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  link_id           INTEGER NOT NULL,             -- 全局有向递增 ID (区分方向)
  src_node          INTEGER NOT NULL,             -- 源节点数字 ID (QNode.json.name)
  src_port          INTEGER NOT NULL,             -- 源端口 (leftLabel)
  dst_node          INTEGER NOT NULL,             -- 目的节点数字 ID
  dst_port          INTEGER NOT NULL,             -- 目的端口 (rightLabel)
  speed             INTEGER NOT NULL,             -- 链路速率 Mbps
  st_queues         INTEGER NOT NULL DEFAULT 3,   -- 源节点队列数 (默认 3)
  PRIMARY KEY (session_id, link_id)
);
CREATE INDEX idx_topo_feature_session_src ON topo_feature_links(session_id, src_node);
CREATE INDEX idx_topo_feature_session_dst ON topo_feature_links(session_id, dst_node);
```

注：`macrotick` 字段在 CDT 代码已注释（util.ts:207/220），不进表；运行时由 `buildDispatchParamsOld` 动态注入。

### 3. `node.json` → 11 张表（base + 10 类业务配置）

`node.json` 顶层结构 `{ <node_id>: {...}, global: {...} }`。`node_id="global"` 作为特殊行。

#### `nodes` (base_info_cfg)
```sql
CREATE TABLE nodes (
  session_id        TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  node_id           TEXT NOT NULL,                -- 数字字符串 或 "global"
  is_global         INTEGER NOT NULL DEFAULT 0,
  node_name         TEXT,
  node_type         TEXT CHECK (node_type IS NULL OR node_type IN ('switch', 'networkcard', 'T10')),
  queue_num         INTEGER DEFAULT 8,
  buffer_num        INTEGER DEFAULT 3,
  port_num          INTEGER DEFAULT 6,
  mac_address       TEXT,                         -- CUC 通信端口 MAC
  ip                TEXT,                         -- CUC 通信端口 IP
  config_file_name  TEXT,
  device_id         TEXT,
  test_port         TEXT,
  PRIMARY KEY (session_id, node_id)
);
```

#### `nodes_oss_cfg` (时钟同步 gPTP)
```sql
CREATE TABLE nodes_oss_cfg (
  session_id        TEXT NOT NULL,
  node_id           TEXT NOT NULL,
  sync_period            TEXT,
  measure_period         TEXT,
  mean_link_delay_thresh TEXT,
  offset_threshold       TEXT,
  -- 其他 oss 字段（per_port master/slave 等）走 JSON
  port_role_assignments  TEXT,                    -- JSON: { "port_0": "master|slave", ... }
  extra_params           TEXT,                    -- JSON: 其余未列字段
  PRIMARY KEY (session_id, node_id),
  FOREIGN KEY (session_id, node_id) REFERENCES nodes(session_id, node_id) ON DELETE CASCADE
);
```

#### `nodes_sdu_table_cfg` (每端口 8 traffic_class 的 SDU 大小)
```sql
CREATE TABLE nodes_sdu_table_cfg (
```

## Outstanding 项需 boss 决策

1. **brainstorm v3 R1 / plan v3 R1 修正**：删除 `data-server.json` / `mac-forwarding-table.json` 引用；加入 `node.json`；明确 P0 范围是"topology + topo_feature + node"3 件套。是否同意？

2. **P0 表数量 6→15**：plan v3 U2a 写的"新增 6 张 topology 表"应改为"新增 15 张 P0 表（含 node.json 派生的 11 张子表）"。是否同意？

3. **node.json 拆分粒度**：每类业务配置（gcl/sdu/psfg/frer/...）独立子表，更利于 SQL 查询和 byte-equal round-trip。**但代价是 15 张表 +大量 FK**。备选方案是把 9 类配置全部作为 `nodes` 表的 JSON 列（简单但失去 SQL 查询能力，违反"消除 JSON blob"主驱动）。当前草案选**子表展开**。是否同意？

4. **`global` 节点处理**：CDT `node.json` 顶层含 `global` 键存全局配置（时钟主节点等）。本草案用 `nodes(node_id='global', is_global=1)` 特殊行。是否同意？或单独 `global_cfg` 表？

5. **byte-equal round-trip 适用范围**：3 件套 (topology + topo_feature + node) JSON 字段平铺到 SQL 表，round-trip 时 JSON 内嵌字段顺序、float 精度、JSON 对象 key 顺序需要 canonical JSON (RFC 8785) 处理。`node.json` 含大量 JSON 内嵌列（如 `nodes_oss_cfg.port_role_assignments`），需要 canonical 序列化保证 byte-equal。是否仍坚持 byte-equal SC 还是改 "结构等价"？

6. **`data-server.json` 处置**：CDT 它是上游 Qunee 编辑器源数据，不在 4 件套内。本 plan 是否 import/export 时也带它？还是认为它只是 CDT 工程的"创作输入"不进我们的 P0 范围？

7. **Spike A 准备**：选定 3 个 fixture（generic-line / generic-ring / dual-plane-redundant）跑 byte-equal round-trip。但我们目前没有 CDT 实际样例的 fixture，需要 boss 提供至少一个 CDT 真实 4 件套样本（小型，3-5 节点）作为 Spike A baseline。是否能提供？
